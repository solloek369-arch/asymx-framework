# Cross-Case Visibility

**Status:** shared pointer layer  
**Function:** make patterns visible across cases without merging claims.  
**Core rule:** overlap is signal, not proof.

This note exists so different public case studies can be seen next to each other without becoming one combined dossier.

A cross-case view may ask:

- Which public signals appear in more than one case?
- Which operational patterns look similar?
- Which questions repeat?
- Which tensions remain case-specific?
- Which source boundaries must stay separate?

## Safe comparison

Comparison may show possible overlap, but it should not claim that one case proves another.

Use language such as:

- possible pattern
- recurring question
- visible similarity
- public signal
- hypothesis
- needs separate validation

## Boundary rule

Each case keeps its own source boundary.

Bol.com does not prove Sligro.

Sligro does not prove Bol.com.

A shared pattern is a pointer for investigation, not a conclusion.

## Minimal AsymX use

AsymX may act here as a Looking Glass:

> a way 2 look at source, signals, friction and overlap without turning the view into proof.

## Safety line

Cross-case visibility should make the field easier to navigate, not heavier to defend.
