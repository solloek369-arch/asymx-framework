# ASYMX Public Data Governance — Sligro Public Source Reviews

## Status

PUBLIC SOURCE REVIEW / OPEN SOURCE VISIBILITY LENS

This directory uses only publicly visible information.

It does not use:
- internal Sligro data
- private communication
- employee reports
- confidential documents
- internal measurements
- operational claims from inside the company

## Purpose

The purpose is to translate public signals into claim-safe AsymX visibility questions and small validation patterns.

This is not:
- an internal case study
- an audit
- a diagnosis
- a consultancy report
- a claim about internal software performance
- a claim about operational causes inside Sligro

## Layer Separation

Every statement must remain traceable to one of three layers:

1. Public source
2. General sector pattern
3. Explicitly marked AsymX interpretation / hypothesis

These layers must not be mixed.

## Aowed Language

Use:

- Public sources describe...
- This publicly visible signal can be read as...
- In logistics operations, this pattern may become visible...
- This is researchable through existing exports...
- This may invite shared validation...
- This is visible enough to review together...
- Based on publicly available information only...

## Forbidden Language

Do not use:

- Sligro has...
- Sligro proves...
- The system fails...
- The floor repairs system errors...
- This proves...
- This guarantees...
- Internal problems exist...
- Root cause is...
- The operation suffers from...

## AC Mix Model

This governance layer uses the AC Mix model as a balancing mechanism.

It keeps the work between two extremes:

1. Too open:
   - public company name without source status
   - interpretation mixed with fact
   - visibility turning into diagnosis

2. Too closed:
   - treating public information as private
   - hiding movement
   - panic cleanup
   - removing useful public-source learning

The middle position is:

- do not hide
- do not overclaim
- do not diagnose
- do not pretend internal access
- do make source status visible
- do separate public source from AsymX interpretation
- do invite challenge, correction and verification

## Change Process

This repository may change when the visibility boundary becomes clearer.

A change is not treated as damage control.

A change is treated as visible calibration.

When direction changes, the reason should be documented:

1. what we looked at first
2. what became visible
3. why the movement changed
4. what the new middle position is
5. what remains open for feedback

## Feedback

AsymX is an open-source framework.

Operational experts, legal reviewers, domain experts and affected parties are invited to challenge, verify or correct our interpretation of public signals via pull request, issue or direct contact.

If something can be improved, that improvement becomes part of the system.

## Required Case Notice

Every case document in this directory should start with:

> **Public Source Notice:** This open-source document translates publicly visible signals into AsymX visibility questions. It relies entirely on public information. It contains no internal company data, performs no operational diagnosis, and makes no claims about internal software performance. See `00_ASYMX_PUBLIC_DATA_GOVERNANCE.md`.

## Closing Rule

Public signal → visible source status → AsymX interpretation → claim boundary → open correction.
