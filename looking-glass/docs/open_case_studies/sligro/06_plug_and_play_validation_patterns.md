# Case 06 — Plug-and-Play Validation Patterns

> **Public Source Notice:** This open-source document translates publicly visible signals into AsymX visibility questions. It relies entirely on public information. It contains no internal company data, performs no operational diagnosis, and makes no claims about internal software performance. See `00_ASYMX_PUBLIC_DATA_GOVERNANCE.md`.

## Purpose

Case 06 translates only visible-enough questions from Case 05 into small, read-only validation patterns.

It does not validate internal truth. It only checks whether a public-source question is structured enough to review safely, locally and without diagnosis.

## What This Is

- Read-only validation layer
- Small reversible check
- Source-to-question-to-pattern bridge
- Routing layer toward Case 07

## What This Is Not

- Diagnosis
- Proof
- Root-cause analysis
- Audit
- Dashboard
- ROI case
- Implementation proposal
- Scanner UI
- Automatic decision system

## Entry Criteria

A question may enter Case 06 only when:

- Public source status is visible
- AsymX interpretation is labeled
- Claim boundary is present
- The question is small enough for a read-only check
- Existing exports or public/logical fields are enough
- No internal diagnosis is made

## Safe Validation Patterns

### 1. Repeated Location Signal Review

- Source signal: repeated location-related questions from Case 05.
- Question from Case 05: is a visible signal repeated enough to review?
- Minimal check: count repeated read-only signals by location or logical area.
- Required data: event type, location or area, timestamp.
- Output: a repeated location signal may be visible enough for review.
- Claim boundary: this does not prove cause or claim internal failure.
- Routing: ROUTE TO CASE 07 or KEEP OPEN.

### 2. Planning Image vs. Execution Signal Review

- Source signal: public planning and execution themes.
- Question from Case 05: can a planning-image question be reviewed without claiming operational truth?
- Minimal check: compare a planned signal with a read-only execution signal.
- Required data: planning field, execution field, timestamp or period.
- Output: a visible difference may indicate a reviewable question.
- Claim boundary: this does not prove planning failure or execution failure.
- Routing: MARK AS ASYMX HYPOTHESIS or KEEP OPEN.

### 3. Legacy-to-Standard Transition Question Review

- Source signal: public ERP transition context.
- Question from Case 05: which transition questions are visible enough to validate safely?
- Minimal check: map public transition language to a small read-only process question.
- Required data: public source, process label, question label.
- Output: a transition question can be reviewed as a visibility question.
- Claim boundary: this does not claim software failure, internal problem or operational cause.
- Routing: ROUTE TO CASE 07 or ASK FOR PUBLIC SOURCE.

### 4. Service Promise Boundary Review

- Source signal: public service, delivery reliability and customer promise themes.
- Question from Case 05: where can a promise-to-signal boundary be reviewed safely?
- Minimal check: compare public promise language with a small visible signal category.
- Required data: public source, promise label, signal category.
- Output: a boundary may be visible enough for shared review.
- Claim boundary: this does not prove service issue or delivery cause.
- Routing: ROUTE TO CASE 07 or KEEP OPEN.

### 5. Visibility Gap Review

- Source signal: a question where public visibility and operational reality are not the same layer.
- Question from Case 05: what remains visible, and what must stay open?
- Minimal check: separate public source, AsymX interpretation and unknowns.
- Required data: source reference, interpretation label, claim boundary.
- Output: the gap can be named without diagnosis.
- Claim boundary: a visibility gap is not proof of internal failure.
- Routing: MARK AS ASYMX HYPOTHESIS or PARK FOR LATER.

### 6. Public-Source Question Routing Review

- Source signal: a public-source question that may be too broad for validation.
- Question from Case 05: should this question move, stay open or park?
- Minimal check: test the question against source status, size and claim boundary.
- Required data: question, public source status, claim boundary.
- Output: the question receives a safe routing status.
- Claim boundary: routing is not conclusion or proof.
- Routing: KEEP OPEN, ASK FOR PUBLIC SOURCE, ROUTE TO CASE 07 or PARK FOR LATER.
